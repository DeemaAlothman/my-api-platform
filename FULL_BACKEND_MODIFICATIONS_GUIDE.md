# القسم الأول: الإصلاحات الأمنية

## 🔴 1.1 إصلاح ثغرة SQL Injection في Auth Service

### المشكلة
في خدمة المصادقة، يتم بناء استعلامات SQL عن طريق إدراج قيم المستخدم (مثل اسم المستخدم وكلمة المرور) مباشرةً داخل نص الاستعلام. هذا يعني أن مستخدماً خبيثاً يستطيع إدخال كود SQL ضمن حقل اسم المستخدم وتنفيذه على قاعدة البيانات.

### الخطورة
- قراءة بيانات كل المستخدمين
- تجاوز المصادقة تماماً
- تدمير أو تعديل البيانات

### الحل المطلوب
يجب استبدال كل الاستعلامات اليدوية (raw SQL) في Auth Service بأساليب Prisma Client المُدمجة (مثل `findUnique`, `findFirst`, `create`)، لأن Prisma يعالج المتغيرات بأمان تلقائياً ولا يسمح بحقن SQL. إذا كان هناك ضرورة قصوى لاستخدام raw queries، يجب استخدام المعاملات المُرقَّمة (parameterized placeholders) بدلاً من إدراج القيم مباشرةً في النص.

---

## 🔴 1.2 إكمال تنفيذ Logout بشكل صحيح

### المشكلة
عند قيام المستخدم بتسجيل الخروج، النظام الحالي لا يُبطل الـ Access Token. يعني ذلك أنه إذا سرق أحدٌ الـ token، يستطيع استخدامه حتى انتهاء صلاحيته (15 دقيقة) حتى بعد تسجيل الخروج.

### الحل المطلوب
إنشاء جدول في قاعدة البيانات يُسمى `token_blacklist` أو `revoked_tokens`، يحتوي على التوكن المُبطل وتاريخ انتهائه. عند كل طلب API، يتحقق النظام أولاً من أن التوكن غير موجود في هذا الجدول قبل قبوله. يجب تنظيف الجدول دورياً من التوكنات المنتهية الصلاحية. كبديل، يمكن الاعتماد على تخزين التوكنات النشطة فقط في Redis مع وقت انتهاء تلقائي.

---

## 🔴 1.3 تأمين رفع الملفات (File Upload Security)

### المشكلة
النظام يخزن روابط للملفات المرفقة في قاعدة البيانات (مرفقات الموظفين، الحيازات، الإجازات، التقييمات)، لكن لا يوجد كود واضح لمعالجة رفع هذه الملفات بشكل آمن.

### الحل المطلوب
يجب تنفيذ نقطة رفع ملفات موحدة (Upload Endpoint) في كل خدمة تحتاجها، مع الشروط التالية:
- **التحقق من نوع الملف:** السماح فقط بأنواع محددة مثل PDF, JPG, PNG, DOCX. رفض أي ملف تنفيذي (exe, js, php...).
- **الحد الأقصى للحجم:** تحديد حجم أقصى لكل ملف (مثلاً 10MB).
- **إعادة تسمية الملفات:** يجب تغيير اسم الملف عند الرفع إلى اسم عشوائي حتى لا يمكن تخمين مسارات الملفات.
- **فصل مجلد الرفع:** يجب تخزين الملفات خارج مجلد الكود (خارج src)، ويفضل في Storage خارجي (S3, MinIO).
- **التحقق من الامتداد والمحتوى:** ليس كافياً التحقق من الامتداد فقط، يجب التحقق من الـ MIME type الفعلي للملف.

## 🔴 1.4 إضافة CSRF Protection

### المشكلة
النظام لا يحمي من هجمات Cross-Site Request Forgery، حيث يمكن لموقع خبيث خداع المستخدم لتنفيذ طلبات باسمه دون علمه.

### الحل المطلوب
إضافة CSRF token في Gateway. عند أول زيارة، يُرسل النظام CSRF token للعميل، ويُطلب منه إرساله مع كل طلب تعديل (POST, PUT, PATCH, DELETE). أو كبديل أبسط: التأكد من أن جميع الـ cookies تحمل خاصية `SameSite=Strict` لمنع الإرسال العابر للمواقع.

---

## 🟠 1.5 إضافة Rate Limiting على جميع الخدمات

### المشكلة
Rate Limiting موجود فقط على نقاط المصادقة (login, refresh). باقي الخدمات (الإجازات، الموظفون، التقييمات، الطلبات) لا تحمي من الإرسال المتكرر والمفرط.

### الحل المطلوب
تطبيق Rate Limiting على مستويين:
1. **مستوى الـ Gateway:** حد عام لكل عنوان IP (مثلاً 100 طلب / دقيقة).
2. **مستوى الخدمة:** حد خاص لكل مستخدم على العمليات الحساسة مثل إنشاء الطلبات والموافقة عليها، بحيث لا يمكن لمستخدم إنشاء أكثر من عدد معين من الطلبات في وقت قصير.

---

## 🟠 1.6 التحقق من الصلاحيات في الوقت الفعلي

### المشكلة
الصلاحيات الحالية مخزنة داخل الـ JWT Token. إذا تم تغيير صلاحيات مستخدم أو إلغاء دوره، يبقى يعمل بصلاحياته القديمة حتى انتهاء مدة التوكن (15 دقيقة).

### الحل المطلوب
على العمليات الحساسة (الحذف، الموافقات، تعديل الرواتب)، يجب التحقق مباشرةً من قاعدة البيانات أن المستخدم لا يزال يملك الصلاحية المطلوبة. يمكن استخدام Redis كـ cache لتسريع هذا التحقق بدلاً من ضرب DB في كل طلب.

---

## 🟠 1.7 حذف نقاط الـ API التجريبية

### المشكلة
يوجد endpoint باسم `_guard-test` في خدمة المستخدمين. هذه النقطة موجودة لأغراض التطوير ولا يجب أن تكون متاحة في البيئة الإنتاجية.

### الحل المطلوب
حذف جميع endpoints التجريبية. وبشكل عام، يجب مراجعة كل الـ routes وإزالة أي شيء موجود للاختبار فقط قبل النشر الإنتاجي.

---

## 🟡 1.8 تحسين إدارة API Keys لأجهزة البصمة

### المشكلة
أجهزة البصمة (ZKTeco) تصادق نفسها باستخدام API Key مخزن في قاعدة البيانات. حالياً لا يوجد آلية لتدوير هذه المفاتيح (key rotation).

### الحل المطلوب
إضافة endpoint مخصص للمسؤولين لإعادة توليد API Key لأي جهاز. تسجيل تاريخ آخر استخدام لكل مفتاح. إضافة تنبيه إذا لم يُستخدم المفتاح لفترة طويلة.

## 🔴 2.1 إكمال نظام الرواتب (Payroll Management)

### الوضع الحالي
يوجد نموذج `MonthlyPayroll` في خدمة الحضور يحسب عدد أيام الحضور والغياب والساعات الإضافية. لكن الحسابات المالية الفعلية غائبة.

### المطلوب إضافته

#### أ. مكوّنات الراتب (Salary Components)
يجب إنشاء جدول يحدد مكونات راتب كل موظف:
- **المكوّنات الإضافية:** بدلات (سكن، مواصلات، هاتف، وجبة، طبيعة عمل)، مكافآت دورية، علاوات.
- **الاستقطاعات الثابتة:** تأمينات اجتماعية (نسبة ثابتة من الراتب الأساسي)، تأمين صحي، استقطاع قرض، سلفة.
- **الاستقطاعات المتغيرة:** ضريبة الدخل (تُحسب تصاعدياً حسب الشريحة الضريبية)، خصم الغياب والتأخير (مرتبط بنتائج الحضور).

#### ب. محرك حساب الراتب (Payroll Engine)
عملية تُشغَّل شهرياً تقوم بـ:
1. جلب راتب الموظف الأساسي من جدول الموظفين.
2. إضافة البدلات المخصصة له.
3. جلب نتائج الحضور من `MonthlyPayroll` وحساب قيمة الخصومات المستحقة.
4. حساب قيمة الساعات الإضافية حسب معدل محدد.
5. تطبيق الاستقطاعات الثابتة (تأمينات، ضريبة).
6. الوصول إلى صافي الراتب المستحق.

#### ج. صلاحيات الرواتب
- مسودة → مراجعة HR → اعتماد المدير المالي → مؤكد للصرف.

#### د. تصدير الرواتب
- تصدير ملف Excel يحتوي على كشف رواتب لكل موظف.
- تصدير ملف بتنسيق البنك (التحويل المباشر).
- كشف راتب فردي للموظف (Payslip).

#### هـ. السجل التاريخي
الاحتفاظ بسجل كامل لكل شهر مع قفله بعد الاعتماد وعدم السماح بالتعديل.

## 🔴 2.2 إدارة المستندات والوثائق (Document Management)

### المشكلة
لا يوجد في النظام أي إدارة مركزية للمستندات الرسمية. المرفقات الموجودة مجرد روابط URL مرتبطة بالكيانات.

### المطلوب إضافته

#### أ. مكتبة المستندات
إنشاء خدمة مستقلة أو وحدة داخل Users Service لإدارة:
- **مستندات الموظف:** عقد العمل، بطاقة الهوية، الشهادات، صور شخصية، وثائق الإقامة.
- **مستندات الشركة:** السياسات الداخلية، اللوائح، دليل الموظف.
- **مستندات الطلبات:** خطابات الموافقة، مذكرات الإجراءات.

#### ب. هيكل المستند
كل مستند يحتوي على: النوع، تاريخ الإنشاء، تاريخ الانتهاء (إن وجد)، مرفوع بواسطة، مرتبط بموظف أو طلب معين، حالة (سارٍ، منتهٍ، ملغى).

#### ج. الصلاحيات
- HR ترفع وتدير.
- الموظف يرى مستنداته فقط.
- المدير يرى مستندات فريقه.

#### د. التنبيهات
إرسال تنبيه قبل انتهاء صلاحية وثيقة (مثل انتهاء الإقامة أو الشهادات) بـ 30 يوماً.

## 🔴 2.3 سير عمل الإلحاق والإنهاء (Onboarding / Offboarding)

### المشكلة
يوجد حقل `probationPeriod` في جدول الموظفين لكن لا توجد أي عمليات مرتبطة به. كذلك لا يوجد سير عمل منظم لاستقبال موظف جديد أو إنهاء خدمة موظف.

### المطلوب إضافته

#### أ. قوالب المهام (Checklist Templates)
جدول يحتوي قوالب مهام قابلة لإعادة الاستخدام:
- قالب الاستقبال (Onboarding): يتضمن مهام مثل "تسليم الجهاز"، "إنشاء حساب البريد"، "إتمام دورة التعريف"، "توقيع العقد".
- قالب الإنهاء (Offboarding): يتضمن مهام مثل "استرداد الحيازات"، "إلغاء الوصول"، "مقابلة الخروج"، "تسوية الراتب النهائي".

#### ب. عملية الإلحاق
عند إضافة موظف جديد، يُنشئ النظام تلقائياً قائمة مهام من القالب المحدد، ويُعيِّن كل مهمة للمسؤول عنها (HR، IT، المدير المباشر). تُتابع نسبة الإنجاز ويُنبَّه المسؤولون على المهام المتأخرة.

#### ج. متابعة فترة التجربة
- تحديد تاريخ انتهاء فترة التجربة تلقائياً من تاريخ التعيين.
- إرسال تنبيه للمدير وHR قبل الانتهاء بأسبوع للبت في تثبيت الموظف أو إنهاء خدمته.
- تسجيل قرار التجربة مع الملاحظات.

#### د. عملية الإنهاء
عند تقديم استقالة أو قرار إنهاء خدمة:
- إنشاء قائمة مهام الإنهاء.
- ربطها بنموذج الطلبات الموجود (طلب استقالة).
- التأكد من إتمام كل الخطوات قبل تحديث حالة الموظف إلى "منتهٍ".

#### هـ. مقابلة الخروج (Exit Interview)
نموذج إلكتروني يُملأ من الموظف يحتوي على أسئلة مقننة عن تجربته وسبب المغادرة. النتائج مجمَّعة لتحليل معدلات الدوران الوظيفي.

## 🔴 2.4 إكمال نظام التوظيف والاستقطاب (Recruitment)

### الوضع الحالي
توجد خدمة `jobs` تحتوي وحدة طلبات التوظيف (Job Applications) لكنها شبه فارغة.

### المطلوب إضافته

#### أ. طلب الاستقطاب (Hiring Request)
- نوع الطلب موجود في Requests Service لكن بدون تفاصيل.
- يجب ربطه بجدول تفاصيلي يحتوي: المسمى الوظيفي المطلوب، القسم، عدد الشواغر، نوع العقد، الراتب المتوقع، تاريخ الحاجة، مبررات الطلب.

#### ب. إدارة المتقدمين (Candidate Pipeline)
جدول المتقدمين يحتوي: الاسم، بيانات التواصل، السيرة الذاتية، المسمى المتقدم له، مصدر الطلب (موقع، تحويل داخلي، LinkedIn).

مراحل التقديم:
1. **جديد:** طلب وارد.
2. **فرز أولي:** مراجعة HR للسيرة الذاتية.
3. **مقابلة هاتفية:** جدولة ومخرجات.
4. **مقابلة تقنية:** جدولة مع المدير، تقييم الكفاءة.
5. **مقابلة نهائية:** مع الإدارة العليا إن لزم.
6. **عرض عمل:** إرسال Offer وانتظار القبول.
7. **مقبول / مرفوض / انسحب.**

#### ج. جدولة المقابلات
تسجيل موعد المقابلة والمحاور والمكان. إرسال دعوة تقويم للمتقدم. تسجيل ملاحظات المقابلة وتقييم المحاور.

#### د. عرض العمل (Offer Letter)
إنشاء عرض عمل بالراتب والمزايا وتاريخ المباشرة. حالة: مُرسَل، مقبول، مرفوض، منتهي الصلاحية.

#### هـ. تحويل المتقدم لموظف
عند القبول: إنشاء حساب مستخدم وسجل موظف مباشرةً من بيانات المتقدم دون إعادة إدخال.

## 🟠 2.5 نظام الإشعارات (Notifications)

### المشكلة
لا يوجد أي آلية لإشعار المستخدمين بالأحداث المهمة. كل الموافقات والطلبات تتطلب من المستخدم التحقق يدوياً.

### المطلوب إضافته

#### أ. قناة الإشعارات
- **بريد إلكتروني:** لكل الأحداث المهمة (طلب يحتاج موافقة، طلب تم قبوله/رفضه).
- **إشعارات داخل التطبيق (In-App):** جرس إشعارات مع عدد الإشعارات غير المقروءة.

#### ب. الأحداث التي تولّد إشعاراً
| الحدث | المستلم |
|-------|---------|
| طلب إجازة جديد | المدير المباشر |
| موافقة/رفض الإجازة | الموظف |
| طلب يحتاج موافقتك | المعتمد المعني |
| اعتماد/رفض الطلب | صاحب الطلب |
| تنبيه حضور (غياب/تأخر) | الموظف والمدير |
| انتهاء فترة التجربة قريباً | HR والمدير |
| انتهاء صلاحية وثيقة | HR |
| دورة تقييم مفتوحة | الموظف المعني |
| تسجيل دخول من جهاز جديد | الموظف نفسه |

#### ج. قاعدة بيانات الإشعارات
جدول يخزن كل إشعار مع: المستلم، النوع، العنوان، المحتوى، رابط الإجراء المطلوب، حالة القراءة، التاريخ.

#### د. إعدادات الإشعارات
كل مستخدم يستطيع اختيار أي الإشعارات يريد استقبالها وعبر أي قناة.

## 🟠 2.6 التقارير والتحليلات (Reports & Analytics)

### المشكلة
لا يوجد نظام تقارير موحد يغطي جميع الخدمات. القرارات الإدارية تتطلب رؤية شاملة للبيانات.

### المطلوب إضافته

#### أ. تقارير الموارد البشرية
- **تقرير الموظفين:** عدد الموظفين حسب القسم، الجنسية، الجنس، نوع العقد، الحالة.
- **تقرير الدوران الوظيفي:** معدل التعيين والإنهاء شهرياً/سنوياً مع أسباب الإنهاء.
- **تقرير الرواتب:** إجمالي الرواتب حسب القسم، متوسط الراتب، نطاق الرواتب.
- **تقرير تواريخ الاستحقاق:** العقود القريبة من الانتهاء، الإقامات المنتهية.

#### ب. تقارير الإجازات
- رصيد الإجازات لكل الموظفين.
- توزيع الإجازات المأخوذة حسب النوع والشهر.
- الإجازات المعلقة والموافق عليها.

#### ج. تقارير الحضور
- معدل الحضور اليومي والشهري.
- قائمة الموظفين الأكثر تأخراً وغياباً.
- الساعات الإضافية المتراكمة.
- ملخص الحضور الشهري لكل موظف.

#### د. تقارير الأداء
- توزيع درجات التقييم.
- مقارنة أداء الأقسام.
- الموظفون الحاصلون على توصيات (ترقية، علاوة، إنذار).

#### هـ. تنسيقات التصدير
كل تقرير يُصدَّر بصيغة Excel و PDF.

## 🟠 2.7 المخطط التنظيمي وهيكل الأقسام (Org Chart API)

### المشكلة
قاعدة البيانات تدعم التسلسل الهرمي للأقسام والتبعية الوظيفية، لكن لا توجد نقطة API تعيد هذه البيانات بتنسيق هيكلي مناسب لرسم المخطط التنظيمي.

### المطلوب إضافته
- **نقطة API للمخطط:** ترجع هيكل الشركة كاملاً على شكل شجرة متداخلة (nested tree)، كل عقدة تحتوي: معلومات القسم، اسم المدير، عدد الموظفين، الأقسام الفرعية.
- **نقطة API للتسلسل الرئاسي:** لأي موظف، تعيد سلسلة المسؤولية من الموظف حتى أعلى المستويات.
- **نقطة API للفريق المباشر:** لأي مدير، تعيد قائمة الموظفين التابعين له مباشرةً.

## 🟠 2.8 نظام Audit Log مركزي

### المشكلة
بعض الخدمات تحتفظ بسجل تغييرات جزئي (مثل `LeaveRequestHistory`)، لكن لا يوجد سجل مركزي موحد يتتبع كل العمليات على مستوى النظام.

### المطلوب إضافته

#### أ. جدول الـ Audit Log
في كل خدمة أو في خدمة مركزية مستقلة:
- **من:** معرف المستخدم.
- **ماذا:** الإجراء (إنشاء، تعديل، حذف، موافقة، رفض...).
- **على ماذا:** اسم الجدول ومعرف السجل.
- **متى:** الطابع الزمني الدقيق.
- **من أين:** عنوان IP، نوع الجهاز.
- **التغيير:** الحالة قبل وبعد التعديل (للعمليات الحساسة).

#### ب. الأحداث الواجب تسجيلها
- تسجيل الدخول والخروج وفشل محاولات الدخول.
- كل عمليات الإنشاء والتعديل والحذف.
- تغيير الصلاحيات.
- الموافقات والرفض على الطلبات.
- عمليات تصدير البيانات.
- تغيير كلمة المرور.

#### ج. خصائص الـ Audit Log
- يُكتب فقط ولا يُعدَّل أبداً (append-only).
- لا يمكن لأي مستخدم حذفه حتى المدير العام.
- محمي من الوصول غير المصرح به.

---

## 🟠 2.9 البوابة الذاتية للموظف (Employee Self-Service)

### المشكلة
الموظف يعتمد كلياً على HR لمعظم العمليات. يجب تمكين الموظف من إدارة ما يخصه مباشرةً.

### المطلوب إضافته

#### أ. عرض وتعديل البيانات الشخصية
- الموظف يرى بياناته الكاملة.
- يستطيع تعديل: رقم الهاتف، العنوان، معلومات جهة الاتصال في الطوارئ، بيانات الحساب البنكي.
- التعديلات تذهب لموافقة HR قبل التطبيق (لا تُطبَّق مباشرة).

#### ب. الخدمات المتاحة
- عرض رصيد الإجازات.
- عرض وتنزيل كشوف الرواتب (Payslips) لكل شهر.
- عرض سجل الحضور الشخصي.
- عرض التقييمات والأهداف.
- عرض طلباته التاريخية وحالتها.
- تغيير كلمة المرور.

---

## 🟠 2.10 نظام التدريب وتطوير الكفاءات (Training Management)

### المشكلة
يوجد حقل لشهادات التدريب في ملف الموظف، لكن لا يوجد نظام لإدارة التدريب كعملية متكاملة.

### المطلوب إضافته

#### أ. كتالوج الدورات التدريبية
جدول يحتوي الدورات المتاحة: الاسم، الوصف، المزود (داخلي/خارجي)، المدة، التكلفة، الكفاءة التي تطورها.

#### ب. طلب وتعيين التدريب
- الموظف يطلب الالتحاق بدورة.
- المدير يوافق أو يرفض.
- HR تعين موظفاً على دورة مباشرةً.
- تسجيل: تاريخ البداية والنهاية، الحالة (مخطط، جارٍ، مكتمل، ملغى).

#### ج. متابعة الإنجاز
- درجة الاجتياز إن وجدت.
- رفع شهادة الإنجاز كمرفق.
- ربط الشهادة بملف الموظف تلقائياً.

#### د. التدريب المطلوب حسب الدور
إمكانية تحديد دورات إلزامية لكل مسمى وظيفي أو قسم، مع تنبيهات للموظفين الذين لم يُكملوا التدريب المطلوب.

# القسم الثالث: تحسينات على الخدمات القائمة

---

## 3.1 تحسينات على Auth Service

### أ. قفل الحساب بعد محاولات فاشلة
بعد عدد محدد من محاولات تسجيل الدخول الفاشلة المتتالية (مثلاً 10 محاولات)، يُقفل الحساب مؤقتاً لـ 15 دقيقة. يُرسل إشعار للمستخدم بريد إلكتروني عند القفل.

### ب. سجل جلسات الدخول
جدول يتتبع جلسات الدخول: الجهاز، المتصفح، عنوان IP، بلد الدخول، الوقت. المستخدم يستطيع رؤية جلساته النشطة وإنهاء أي منها عن بُعد.

### ج. تغيير كلمة المرور
- نقطة API لتغيير كلمة المرور تتطلب كلمة المرور الحالية.
- نقطة API لإعادة تعيين كلمة المرور عبر البريد الإلكتروني (forgot password).
- إبطال كل الـ refresh tokens النشطة عند تغيير كلمة المرور.

---

## 3.2 تحسينات على Users Service

### أ. إدارة الموظفين المتقدمة
- **الترقية والنقل:** تسجيل تاريخ كامل للمسميات الوظيفية والأقسام التي مر بها الموظف مع تاريخ كل تغيير.
- **الهيكل العائلي:** إضافة/تعديل/حذف أفراد العائلة المسجلين للموظف.
- **البيانات البنكية:** حفظ بيانات الحساب البنكي للموظف لأغراض الرواتب (مشفرة في قاعدة البيانات).

### ب. إدارة الحيازات (Custody) - تحسينات
- إضافة: تاريخ الإصدار الفعلي وتاريخ الاسترداد المتوقع.
- تنبيه تلقائي عند حلول موعد استرداد الحيازة.
- ربط الحيازة بمهمة الإنهاء (يجب استرداد الحيازات قبل إتمام الإنهاء).
- تقرير بالحيازات المفقودة أو التالفة.

### ج. الهياكل الوظيفية
- ربط الموظف بالدرجة الوظيفية (Job Grade) والتأكد أن راتبه ضمن نطاق الدرجة.
- تنبيه عند وجود موظفين بنفس المسمى دون درجة وظيفية.

---

## 3.3 تحسينات على Leave Service

### أ. الاعتذار (Compensation) عن الإجازات
إذا عمل موظف في إجازة رسمية أو عطلة نهاية الأسبوع، تُسجَّل ساعات العمل الإضافية وتُضاف كأيام إجازة تعويضية.

### ب. تفويض الموافقة (Delegation)
المدير يستطيع تفويض صلاحية الموافقة على الإجازات لمدير آخر خلال فترة غيابه.

### ج. تقارير الإجازات
- تنبيه عند تراكم الإجازات غير المأخوذة لموظف معين فوق حد معين.
- تقرير للإدارة يوضح توزيع الغياب المتوقع خلال الأشهر القادمة بناءً على الإجازات المعتمدة.

---

## 3.4 تحسينات على Attendance Service

### أ. تكامل أفضل مع الرواتب
التأكد من أن نتائج `MonthlyPayroll` مغلقة (locked) بعد اعتماد الرواتب ولا يمكن تعديلها.

### ب. الغياب التلقائي
نظام يعمل بنهاية كل يوم عمل ويتحقق: هل سجّل الموظف حضوره؟ إذا لم يسجل وليس على إجازة مسجلة، يُنشئ سجل غياب تلقائياً وينبّه المدير.

### ج. الأماكن الجغرافية (Geo-fencing) - اختياري
إذا كان الحضور يتم عبر التطبيق، التحقق من أن الموظف داخل نطاق جغرافي محدد (مثلاً نطاق 200 متر من مقر الشركة).

---

## 3.5 تحسينات على Evaluation Service

### أ. ربط التقييم بالترقية والمكافآت
عند إغلاق دورة التقييم، إنشاء قائمة بالموظفين الحاصلين على توصيات ترقية أو علاوة، وربطها بنظام الطلبات للمتابعة الفعلية.

### ب. تقييم 360 درجة الكامل
التأكد من أن التقييم يشمل: الموظف نفسه، المدير المباشر، زملاء من نفس الدرجة، مرؤوسون (إن وجدوا).

### ج. لوحة تحليلات الأداء
- توزيع الدرجات (كم موظف ممتاز / جيد جداً...).
- مقارنة بين الأقسام.
- تتبع تطور أداء الموظف عبر الدورات.

---

## 3.6 تحسينات على Requests Service

### أ. قوالب الطلبات المتخصصة
كل نوع طلب يحتاج جدول تفاصيل خاص به:
- طلب مأمورية: الوجهة، الغرض، المدة، التكلفة المتوقعة.
- طلب تفويض: الصلاحيات المفوَّضة، الفترة الزمنية.
- طلب مكافأة: نوع المكافأة، قيمتها المقترحة، المبررات.
- طلب عقوبة: نوع المخالفة، المادة النظامية، نوع العقوبة المقترحة.

### ب. إشعارات متقدمة
- تذكير تلقائي للمعتمد بعد 48 ساعة من إحالة الطلب إذا لم يتخذ إجراء.
- تصعيد للمستوى الأعلى بعد 72 ساعة.

### ج. إحصائيات الطلبات
- معدل قبول/رفض لكل نوع طلب.
- متوسط وقت البت في الطلبات.
- الطلبات المعلقة أكثر من أسبوع.

# القسم الرابع: البنية التقنية والأداء

---

## 4.1 التواصل بين الخدمات (Inter-Service Communication)

### المشكلة
حالياً يبدو أن كل خدمة مستقلة وليس واضحاً كيف تتبادل البيانات. مثلاً: Attendance تحتاج بيانات الموظفين، Evaluation تحتاج ربط الأهداف بالتقييم.

### المطلوب
- توثيق وتوحيد آلية التواصل: هل HTTP مباشر، أم Message Queue (RabbitMQ / Kafka)؟
- للعمليات المتزامنة: HTTP مع retry logic.
- للأحداث غير المتزامنة (مثل إرسال الإشعارات): يُفضَّل Message Queue.
- في جميع الأحوال: ما يحدث إذا خدمة معطلة يجب أن يكون مُعرَّفاً (fallback behavior).

---

## 4.2 إضافة طبقة Caching

### المشكلة
بيانات كثيرة تُقرأ بشكل متكرر ولا تتغير كثيراً: قائمة الأقسام، أنواع الإجازات، الأدوار والصلاحيات.

### المطلوب
إضافة Redis لتخزين هذه البيانات مؤقتاً. عند تعديل أي منها، يُمسح الـ cache المقابل. هذا يقلل الضغط على قاعدة البيانات بشكل كبير.

---

## 4.3 توثيق API (Swagger / OpenAPI)

### المشكلة
لا يوجد توثيق API آلي. يعتمد الفريق على ملفات Markdown يدوية قد لا تكون محدثة.

### المطلوب
تفعيل Swagger في كل خدمة. كل endpoint يجب أن يكون موثقاً: المعاملات، الاستجابات المتوقعة (ناجحة وخاطئة)، الصلاحيات المطلوبة. يكون Swagger محمياً بكلمة مرور في بيئة الإنتاج.

---

## 4.4 اختبارات تلقائية (Automated Testing)

### المشكلة
لا تظهر ملفات اختبارات في المشروع. هذا خطر كبير عند إجراء تعديلات.

### المطلوب
- **Unit Tests:** لكل service وكل دالة حساب (الرواتب، التحقق من الصلاحيات).
- **Integration Tests:** لكل workflow كامل (تقديم إجازة → موافقة → خصم من الرصيد).
- **E2E Tests:** لأهم السيناريوهات من بداية الطلب حتى النهاية.
- **لا يجوز الـ Mock لقاعدة البيانات** في Integration Tests؛ يجب استخدام قاعدة بيانات اختبار حقيقية لضمان صحة النتائج.
