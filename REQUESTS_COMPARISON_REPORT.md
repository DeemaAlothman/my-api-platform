# تقرير مقارنة الطلبات المطلوبة مع النظام الحالي

---

## النظام الحالي - ملخص

### خدمة الطلبات (Requests Service)
- **أنواع الطلبات الحالية:** TRANSFER, PERMISSION, ADVANCE, RESIGNATION, JOB_CHANGE, RIGHTS, REWARD, SPONSORSHIP, OTHER
- **مسار الاعتماد:** مسودة ← المدير المباشر ← الموارد البشرية ← معتمد/مرفوض (مرحلتين فقط)
- **البيانات:** نوع الطلب + سبب + ملاحظات + مرفق واحد + حقل `details` (JSON مرن)
- **التتبع:** سجل تاريخي كامل (RequestHistory)

### خدمة الإجازات (Leave Service) - منفصلة
- **أنواع الإجازات:** قابلة للتعريف (LeaveType) مع إعدادات لكل نوع
- **أرصدة الإجازات:** نظام أرصدة سنوية لكل موظف ونوع إجازة
- **طلب الإجازة:** يتضمن تاريخ بداية/نهاية، عدد الأيام، نصف يوم، بديل، مسار اعتماد خاص

## المقارنة التفصيلية - الطلبات العشرة

---

### 1. طلب إجازة يومية

| البند | الحالة | التفاصيل |
|-------|--------|----------|
| **موجود؟** | نعم - في خدمة الإجازات | خدمة Leave منفصلة بالكامل |
| **بيانات الموظف** | موجود جزئياً | الطلب يحفظ `employeeId` فقط، بيانات الموظف تُجلب عبر cross-schema |
| **تفاصيل الإجازة** | موجود | `startDate`, `endDate`, `totalDays`, `isHalfDay` |
| **نوع الإجازة** | موجود لكن يحتاج تحديث | الأنواع قابلة للتعريف من LeaveType - يجب إضافة الأنواع المطلوبة |
| **التفويض** | غير موجود | يوجد حقل `substituteId` (البديل) لكن لا يوجد ربط بطلب تفويض |
| **الاعتمادات** | موجود | مسار: مدير مباشر ← HR |

#### التعديلات المطلوبة:
- **إضافة أنواع إجازات جديدة** في جدول `LeaveType`: وفاة، زواج، صحي، أمومة، إداري، بلا راتب
- **ربط بطلب التفويض** (اختياري): إضافة حقل `delegationRequestId` على `LeaveRequest` للربط بطلب تفويض إن وجد
- **لا حاجة لتعديلات كبيرة** - النظام الأساسي جاهز

---

### 2. طلب استقالة

| البند | الحالة | التفاصيل |
|-------|--------|----------|
| **موجود؟** | نعم جزئياً | نوع `RESIGNATION` موجود في خدمة الطلبات |
| **رقم الطلب** | موجود | `requestNumber` يتولد تلقائياً |
| **تاريخ التقديم** | موجود | `createdAt` |
| **بيانات الموظف** | موجود | يُجلب عبر `employeeId` |
| **تاريخ سريان الاستقالة** | غير موجود كحقل مستقل | يمكن تخزينه في `details` (JSON) لكن الأفضل حقل مخصص |
| **الأسباب** | موجود | حقل `reason` |
| **شرط 30 يوم** | غير موجود | لا يوجد validation على فترة الإخطار |
| **سلسلة الاعتمادات** | ناقص | المسار الحالي: مدير ← HR فقط. **المطلوب:** مدير مباشر ← مدير HR ← مدير تنفيذي (3 مراحل) |

#### التعديلات المطلوبة:
- **إضافة حقل `effectiveDate`** على جدول Request (أو استخدام `details` JSON)
- **إضافة validation** للتحقق أن تاريخ السريان بعد 30 يوم على الأقل من تاريخ التقديم
- **إضافة مرحلة اعتماد ثالثة** (المدير التنفيذي) - وهذا تعديل جوهري على مسار الاعتماد بالكامل

---

### 3. طلب اقتراح عقوبة

| البند | الحالة | التفاصيل |
|-------|--------|----------|
| **موجود؟** | لا | لا يوجد نوع طلب للعقوبات |
| **الأطراف** | - | يجب تحديد الموظف المعني بالعقوبة (غير مقدم الطلب) |
| **أسباب العقوبة** | - | يمكن استخدام حقل `reason` الموجود |
| **آراء الإدارة** | - | يحتاج مراحل اعتماد متعددة |
| **القرار النهائي** | - | يحتاج حقول إضافية للعقوبة المقررة |

#### التعديلات المطلوبة:
- **إضافة نوع `PENALTY_PROPOSAL`** في enum `RequestType`
- **إضافة حقل `targetEmployeeId`** على جدول Request (الموظف المعاقب ≠ مقدم الطلب)
- **استخدام `details` JSON** لتخزين: نوع العقوبة المقترحة، العقوبة النهائية المقررة، سبب القرار
- **إضافة مرحلة المدير التنفيذي** في مسار الاعتماد

---

### 4. طلب نقل

| البند | الحالة | التفاصيل |
|-------|--------|----------|
| **موجود؟** | نعم جزئياً | نوع `TRANSFER` موجود |
| **بيانات الموظف الحالية** | موجود | تُجلب عبر `employeeId` (القسم، الراتب، تاريخ التعيين) |
| **تفاصيل النقل** | غير مهيكل | حالياً تُخزن في `details` JSON بشكل حر |
| **الاعتمادات** | ناقص | المسار الحالي: مدير ← HR. المطلوب: المسؤول الحالي + المسؤول المنقول إليه |

#### التعديلات المطلوبة:
- **هيكلة `details` JSON** ليحتوي على: `fromDepartmentId`, `toDepartmentId`, `fromJobTitleId`, `toJobTitleId`, `newSalary`
- أو **إنشاء جدول مخصص `TransferDetails`** مرتبط بالطلب
- **إضافة validation** عند الموافقة لتنفيذ النقل الفعلي (تحديث بيانات الموظف)
- **تعديل مسار الاعتماد** ليشمل المسؤول المنقول إليه

---

### 5. طلب تكليف بعمل إضافي

| البند | الحالة | التفاصيل |
|-------|--------|----------|
| **موجود؟** | لا | لا يوجد نوع طلب للعمل الإضافي |
| **التوقيت** | - | يوم التكليف، ساعات البدء/الانتهاء، عدد الساعات |
| **المهام** | - | وصف الأعمال المطلوبة |
| **شرط التقديم** | - | يجب التقديم قبل 12:00 ظهراً |

#### التعديلات المطلوبة:
- **إضافة نوع `OVERTIME`** في enum `RequestType`
- **استخدام `details` JSON** لتخزين: `overtimeDate`, `startTime`, `endTime`, `totalHours`, `tasks`
- **إضافة validation** للتحقق أن الطلب مقدم قبل الساعة 12:00 ظهراً من يوم التكليف
- مسار الاعتماد الحالي (مدير ← HR) **كافي** لهذا الطلب

### 6. طلب مهمة عمل

| البند | الحالة | التفاصيل |
|-------|--------|----------|
| **موجود؟** | لا | لا يوجد نوع طلب لمهام العمل |
| **نوع المهمة** | - | داخلية / خارجية |
| **المدة والوجهة** | - | تاريخ بداية/نهاية، عدد الأيام، الجهة |
| **الاعتمادات** | - | مسؤول مباشر ← HR ← مدير تنفيذي (3 مراحل) |

#### التعديلات المطلوبة:
- **إضافة نوع `BUSINESS_TRIP`** في enum `RequestType`
- **استخدام `details` JSON** لتخزين: `missionType` (INTERNAL/EXTERNAL), `destination`, `startDate`, `endDate`, `totalDays`, `purpose`
- **يحتاج مرحلة المدير التنفيذي** (تعديل مسار الاعتماد)

---

### 7. طلب تفويض

| البند | الحالة | التفاصيل |
|-------|--------|----------|
| **موجود؟** | لا | لا يوجد نوع طلب للتفويض |
| **نطاق التفويض** | - | كامل / جزئي |
| **الفترة** | - | تاريخ بداية ونهاية |
| **الأطراف** | - | المفوِّض والمفوَّض له |

#### التعديلات المطلوبة:
- **إضافة نوع `DELEGATION`** في enum `RequestType`
- **إضافة حقل `delegateToEmployeeId`** أو استخدام `details` JSON لتخزين: `delegationType` (FULL/PARTIAL), `delegateToId`, `startDate`, `endDate`, `scope` (وصف الصلاحيات المفوضة)
- مسار الاعتماد الحالي **كافي**

### 8. طلب موظف (احتياج توظيف)

| البند | الحالة | التفاصيل |
|-------|--------|----------|
| **موجود؟** | لا | لا يوجد نوع طلب لاحتياج التوظيف |
| **المواصفات** | - | القسم، المسمى، العدد، الجنس، المستوى العلمي، الخبرة |
| **المبررات** | - | سبب الحاجة |
| **الاعتمادات** | - | مسؤول مباشر ← HR ← مدير تنفيذي (3 مراحل) |

#### التعديلات المطلوبة:
- **إضافة نوع `HIRING_REQUEST`** في enum `RequestType`
- **استخدام `details` JSON** لتخزين: `departmentId`, `jobTitleId`, `quantity`, `gender`, `educationLevel`, `experienceYears`, `justification`
- **يحتاج مرحلة المدير التنفيذي**

### 9. طلب اقتراح مكافأة

| البند | الحالة | التفاصيل |
|-------|--------|----------|
| **موجود؟** | نعم جزئياً | نوع `REWARD` موجود |
| **جدول المكافآت** | غير مهيكل | حالياً تُخزن في `details` JSON بشكل حر |
| **الاعتمادات المالية** | غير موجود | لا يوجد مسار اعتماد يشمل المدير المالي |

#### التعديلات المطلوبة:
- **هيكلة `details` JSON** ليحتوي على: `targetEmployeeId`, `rewardType`, `amount`, `currency`, `reason`
- أو دعم **مكافآت جماعية** بمصفوفة موظفين
- **إضافة مرحلة اعتماد مالية** (المدير المالي/المحاسب) - تعديل جوهري على مسار الاعتماد

---

### 10. طلب شكوى

| البند | الحالة | التفاصيل |
|-------|--------|----------|
| **موجود؟** | لا | لا يوجد نوع طلب للشكاوى |
| **المحتوى** | - | تفاصيل الشكوى |
| **التحقيق** | - | أسماء أشخاص للتواصل معهم |
| **الاعتماد** | - | مقدم الشكوى ← مدير HR |

#### التعديلات المطلوبة:
- **إضافة نوع `COMPLAINT`** في enum `RequestType`
- **استخدام `details` JSON** لتخزين: `complaintDetails`, `involvedPersons` (مصفوفة أسماء), `witnesses`
- مسار الاعتماد: يمكن تبسيطه إلى **HR فقط** (تخطي المدير) أو إبقاء المسار الحالي

## التعديلات الجوهرية المشتركة

### 1. توسيع مسار الاعتماد (الأهم والأكبر)

**المشكلة:** المسار الحالي يدعم مرحلتين فقط (مدير ← HR).
**المطلوب:** عدة طلبات تحتاج 3 مراحل أو أكثر:

| المرحلة | الطلبات التي تحتاجها |
|---------|---------------------|
| المدير المباشر | جميع الطلبات |
| مدير الموارد البشرية | أغلب الطلبات |
| المدير التنفيذي | استقالة، عقوبة، مهمة عمل، احتياج توظيف |
| المدير المالي | مكافأة |

**الحل المقترح:** تحويل مسار الاعتماد من حقول ثابتة إلى **جدول اعتمادات ديناميكي** `RequestApproval`:
- كل طلب يمكن أن يكون له عدة مراحل اعتماد
- كل مرحلة لها: ترتيب (`order`)، نوع المعتمد (`approverRole`)، حالة (`status`)، معتمِد (`approvedBy`)
- يسمح بتخصيص مسار اعتماد مختلف لكل نوع طلب

**الملفات المتأثرة:**
- `apps/requests/prisma/schema.prisma` - إضافة model `RequestApproval` وحذف الحقول الثابتة
- `apps/requests/src/requests/requests.service.ts` - إعادة كتابة منطق الاعتماد بالكامل
- `apps/requests/src/requests/requests.controller.ts` - تعديل endpoints الاعتماد
- جميع DTOs المتعلقة بالاعتماد

### 2. إضافة أنواع الطلبات الجديدة

**التعديل على enum `RequestType`:**
```
الحالي:  TRANSFER, PERMISSION, ADVANCE, RESIGNATION, JOB_CHANGE, RIGHTS, REWARD, SPONSORSHIP, OTHER
المطلوب: TRANSFER, RESIGNATION, REWARD (موجودين)
         + OVERTIME (عمل إضافي)
         + BUSINESS_TRIP (مهمة عمل)
         + DELEGATION (تفويض)
         + HIRING_REQUEST (احتياج توظيف)
         + PENALTY_PROPOSAL (اقتراح عقوبة)
         + COMPLAINT (شكوى)
```

### 3. إضافة حقل الموظف المستهدف

بعض الطلبات تستهدف موظفاً آخر (غير مقدم الطلب):
- طلب عقوبة → الموظف المعاقب
- طلب مكافأة → الموظف المكافأ
- طلب تفويض → المفوض له

**الحل:** إضافة حقل `targetEmployeeId` اختياري على جدول Request.

### 4. حقل `details` (JSON) - هيكلة حسب النوع

النظام الحالي يملك حقل `details Json?` مرن. **هذا ممتاز** لأنه يسمح بتخزين بيانات مختلفة لكل نوع طلب بدون تغيير الجدول.

**المطلوب:** إضافة validation على مستوى الـ Service حسب نوع الطلب:

| النوع | حقول details المطلوبة |
|-------|----------------------|
| RESIGNATION | `effectiveDate` |
| TRANSFER | `fromDepartmentId`, `toDepartmentId`, `fromJobTitleId`, `toJobTitleId`, `newSalary` |
| OVERTIME | `overtimeDate`, `startTime`, `endTime`, `totalHours`, `tasks` |
| BUSINESS_TRIP | `missionType`, `destination`, `startDate`, `endDate`, `totalDays` |
| DELEGATION | `delegationType`, `delegateToId`, `startDate`, `endDate`, `scope` |
| HIRING_REQUEST | `departmentId`, `jobTitleId`, `quantity`, `gender`, `educationLevel`, `experienceYears` |
| PENALTY_PROPOSAL | `proposedPenalty`, `finalPenalty`, `penaltyReason` |
| COMPLAINT | `complaintDetails`, `involvedPersons`, `witnesses` |
| REWARD | `rewardType`, `amount`, `currency` |

## ملخص جدول المقارنة

| # | الطلب | موجود؟ | جاهزية النظام | حجم التعديل |
|---|-------|--------|--------------|-------------|
| 1 | إجازة يومية | نعم (خدمة منفصلة) | 85% | صغير - إضافة أنواع إجازات |
| 2 | استقالة | جزئياً | 60% | متوسط - validation + مرحلة اعتماد ثالثة |
| 3 | اقتراح عقوبة | لا | 30% | متوسط - نوع جديد + targetEmployee + مرحلة ثالثة |
| 4 | نقل | جزئياً | 50% | متوسط - هيكلة details + تنفيذ النقل الفعلي |
| 5 | عمل إضافي | لا | 20% | صغير-متوسط - نوع جديد + validation الوقت |
| 6 | مهمة عمل | لا | 20% | متوسط - نوع جديد + مرحلة ثالثة |
| 7 | تفويض | لا | 20% | متوسط - نوع جديد + targetEmployee |
| 8 | احتياج توظيف | لا | 20% | متوسط - نوع جديد + مرحلة ثالثة |
| 9 | مكافأة | جزئياً | 40% | متوسط - هيكلة + مرحلة مالية |
| 10 | شكوى | لا | 20% | صغير - نوع جديد + details |

## الترتيب المقترح للتنفيذ

### المرحلة 1 - الأساس (الأولوية القصوى)
1. **تعديل مسار الاعتماد** ليكون ديناميكي (3+ مراحل) - هذا يؤثر على كل الطلبات
2. **إضافة حقل `targetEmployeeId`** على جدول Request

### المرحلة 2 - الطلبات السهلة
3. إضافة أنواع الإجازات المطلوبة (بيانات فقط في جدول LeaveType)
4. طلب الشكوى (بسيط)
5. طلب العمل الإضافي (بسيط مع validation)

### المرحلة 3 - الطلبات المتوسطة
6. طلب الاستقالة (validation + effectiveDate)
7. طلب التفويض
8. طلب مهمة العمل

### المرحلة 4 - الطلبات المعقدة
9. طلب النقل (يحتاج تنفيذ فعلي لنقل الموظف)
10. طلب المكافأة (مرحلة مالية إضافية)
11. طلب اقتراح العقوبة
12. طلب احتياج التوظيف
